# Аудит приложения «Ставь Угольки»

Проверены зоны:
- `apps/shop` — клиентский mini-app
- `apps/owner` — owner/admin кабинет
- `server` — API, статика, хранение JSON, авторизация
- `bot` — Telegram bot
- `shop-sw.js` — service worker / PWA

## Что было проверено
- синтаксис JS файлов (`node --check`)
- ручной аудит клиентской логики магазина
- ручной аудит owner-панели
- ручной аудит API и хранения данных
- реальные HTTP-проверки сценариев: login, bootstrap, create order, stock decrement, CRUD owner, order history, status update

## Критические проблемы, найденные в коде

### 1) Подмена заказа с клиента
**Было:** `/api/shop/orders` доверял данным из фронта:
- можно было отправить несуществующий товар
- можно было занизить `price`
- можно было занизить `total`
- можно было заказать больше остатка

**Исправлено:** сервер теперь сам:
- сверяет товар с текущим каталогом
- сверяет variant
- проверяет остаток
- пересчитывает цену по каталогу
- пересчитывает total на сервере

### 2) Любой статус заказа принимался сервером
**Было:** `/api/owner/orders/:id` принимал любую строку статуса (`"weird"`, `"abc"`)

**Исправлено:** оставлены только:
- `new`
- `paid`
- `done`
- `cancelled`

### 3) Некорректный расчёт low stock на dashboard
**Было:** summary считал low-stock только по `product.stock`, что давало неверную картину для товаров с вариантами

**Исправлено:** dashboard теперь считает low-stock через `collectLowStockAlerts()`

### 4) Пустые support-контакты можно было сохранить через API
**Было:** owner UI требовал ссылку, но сервер всё равно принимал пустой `link`

**Исправлено:** сервер теперь валидирует `link` на POST/PUT

### 5) HTML-инъекция через `product.image` в shop
**Было:** `product.image` вставлялся в `src` без escaping в карточке и product sheet

**Исправлено:** `src` экранируется

## Риски, которые остаются и требуют внимания

### A) `/api/internal/telegram-config` открыт без секрета, если не задан `CONFIG_SYNC_SECRET`
Это подтверждено реальным запросом к API.

Риск:
- любой, кто знает адрес сервера, может читать/менять chat ids для заявок и постов

Что нужно сделать на Railway:
- обязательно задать `CONFIG_SYNC_SECRET` и одинаковое значение в server и bot

### B) Стандартные логин/пароль owner и стандартный JWT secret
Сейчас код допускает значения по умолчанию.

Риск:
- при забытых env owner-панель легко компрометировать
- подпись токенов предсказуемая при стандартном `JWT_SECRET`

Что нужно сделать:
- задать собственные `OWNER_LOGIN`
- задать собственные `OWNER_PASSWORD`
- задать собственный `JWT_SECRET`

### C) Хранение owner token в `localStorage`
Это рабочий вариант, но он остаётся чувствительным к XSS.
После исправления явной инъекции риск уменьшен, но архитектурно лучше держать токен в httpOnly cookie, если будете делать более защищённую версию.
